### 原始需求（逐字记录）

**初始表述：**
> "我想要做一个工程，团队内成员通过类似于 npm 依赖包安装，初始化后写入 mcp server，然后可以针对各类 IDE 写入针对于项目可用的 agent/rules/skills/mcp 组合，提高开发效率。"

> "建一个自适应的 AI 辅助研发环境，实现 Vue 2/3 混合技术栈的'零配置'接入，并引导前端团队平滑进阶全栈。Core Principle: Specification-Driven Development (SDD)。使用 OpenSpec 作为真理源，MCP 作为上下文传输枢纽。"

**详细补充（2026-01-17）：**

> 项目产生的原因是：当前全球顶尖公司的 IDE、CLI、VSCode 插件/代码助手，都基本支持配置 agent、mcp server、skills、rules 这些工具或配置。但针对团队共同维护项目、以及成员使用平台/插件不一致的情况，想做一个**跨平台的工程**，统一配置，帮助提升团队开发效率。

## 2. 项目定义

### 一句话描述
**跨平台的 AI 辅助研发配置分发工程**：让团队成员无论使用什么 IDE/工具，都能通过一条命令获得统一的、针对当前项目定制的 AI 辅助能力（agent/rules/skills/mcp）。

### 核心目标
1. **跨平台统一**：一套配置，多 IDE 可用（VS Code、Cursor、Claude Code、JetBrains 等）
2. **项目感知**：自动检测项目技术栈，生成针对性的 rules 配置
3. **零配置接入**：团队成员执行固定命令即可快速初始化
4. **渐进式能力**：skills/agent 可预置但不强制使用，用到时再调用

### 非目标（一期明确不做）
- 后端/全栈完整实现（但架构需预留扩展性）
- 跨 IDE 软链接配置共享（前期可不实现）
- 自定义 MCP server 开发（使用现有方案）

## 3. 背景与动机

### 当前痛点
- **配置碎片化**：团队成员使用不同的 IDE/插件，AI 辅助配置各自为政
- **重复劳动**：每个人都要单独配置 agent/rules/mcp，且配置质量参差不齐
- **项目上下文丢失**：AI 助手不了解项目技术栈和约定，每次都要重复解释
- **知识不同步**：一个人发现的好用配置，难以快速同步给团队

### 期望改变
- **一次配置，团队共享**：项目级的 AI 配置可以像代码一样版本管理、分发
- **AI 理解项目**：通过自动检测和生成，让 AI 从第一天就"懂"项目
- **降低门槛**：新成员加入时，执行一条命令即可获得完整的 AI 辅助环境

## 5. 用户场景（Human 描述）

### 核心场景：团队成员初始化
```
场景：新成员加入项目 / 现有成员首次配置

1. 成员执行固定命令（如 npx ai-pilot init）
2. 工程自动检测当前项目技术栈
   - Vue 版本（2/3/混合）
   - 状态管理（Vuex/Pinia）
   - 构建工具（Vite/Vue CLI）
   - TypeScript 配置
3. 根据检测结果生成定制化的 rules
4. 将配置写入对应 IDE 的配置目录
5. Skills/Agent 预置，但不强制启用
```

### 配置分层逻辑
| 类型 | 特点 | 生成策略 |
|------|------|----------|
| **Rules** | 项目强相关 | 根据技术栈检测结果**定制生成** |
| **Skills** | 通用能力 | **预置**，按需调用 |
| **Agent** | 工作流编排 | **预置**，按需调用 |
| **MCP** | 能力接口 | 统一配置 |

### 未来场景（非一期）
- 跨 IDE 配置同步（软链接方案）
- 全栈项目支持（Node.js BFF）
- 团队配置中心化管理

## 6. 成功标准

### MVP 成功的定义
- [ ] 团队成员能通过一条命令完成初始化
- [ ] 工程能正确检测 Vue 2/3 技术栈
- [ ] 生成的 rules 对当前项目有针对性
- [ ] 至少支持 2 种主流 IDE（如 VS Code + Cursor）
- [ ] 团队愿意在日常开发中使用

### 长期成功的定义
- [ ] 成为团队 AI 辅助开发的标准入口
- [ ] 配置可以像代码一样 review 和迭代
- [ ] 新成员 onboarding 时间显著缩短
- [ ] 支持全栈项目
